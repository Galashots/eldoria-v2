## 2. Pipeline bug found and fixed during this task

The first attempt to run `scripts/upscale-nearest-neighbor.mjs` on this master produced a **corrupted** canonical source: **1,044,480 of 1,048,576 output pixels (99.6%) mismatched** their expected source block.

**Root cause:** `readPng()` (in `normalize-asset-sheet.mjs`) always expands its returned `data` buffer to 4 bytes/pixel (RGBA), synthesizing `alpha = 255` for RGB (colorType 2) sources — this is true for every caller, including the approved-master read here. `upscaleNearestNeighborRgbOnly()` computed its read stride as `srcBpp = image.colorType === 6 ? 4 : 3`, incorrectly assuming a colorType-2 source's `image.data` was still packed at 3 bytes/pixel. Because `dirt_center`'s master was colorType 6 (RGBA), `srcBpp` happened to evaluate to 4 there and the bug was latent. `water_a`'s master is colorType 2 (RGB, no alpha channel) — the first Approved-Runtime-Master input to exercise the colorType-2 path — which triggered a one-channel misalignment: e.g. the second output block read `[255, 10, 96]` (the master's alpha byte leaking into red, with green/blue pulled from the next pixel) instead of the correct `[10, 96, 147]`.

**Fix:** removed the colorType-conditional stride; the read stride is now unconditionally 4, matching `readPng()`'s actual always-RGBA-expanded output. Applied directly to `scripts/upscale-nearest-neighbor.mjs` (not routed through a review-only copy, since it is a shared pipeline utility used by every future Approved-Runtime-Master asset).

**Regression coverage:** added a new test block to `scripts/test-asset-pipeline.mjs` that builds a 3×1 colorType-2 source with three distinct, non-repeating pixel colors, upscales it, and asserts each output block starts at the correct source pixel with no channel shift (this exact case would have failed before the fix and passes after). `npm run test:asset-pipeline` passes with the new test included.

After the fix, the upscale was re-run from the same approved master and re-verified: **0 mismatched pixels out of 1,048,576** (see table above).

## 4. Review-only normalization strategy

Everything in this section lives under `docs/art-pipeline/review/tile_farm_water_base_water_a/` — **not** under `assets/manifests/` or `assets/tilesets/` — specifically so it cannot be mistaken for the production manifest/sheet. It exercises the real pipeline scripts (`normalize-asset-sheet.mjs`, `validate-asset-sheet.mjs`), the same ones used for `grass_a` and `dirt_center`, not a parallel implementation.

Settings used, matching Category A conventions:

- background mode: `alpha` (preserve source)
- trim: `none`
- anchor: `top_left`
- fit: `fill`

### Round-trip verification: zero drift

The review manifest normalizes the 1024×1024 canonical source directly back down to 16×16 via the pipeline's real `fit: "fill"` nearest-neighbour scaling. Because the source is an exact 64× block-replication of the approved master, this round trip (16×16 → 1024×1024 → 16×16) is expected to be lossless — and it is: the normalized output's RGB values were compared pixel-by-pixel against the original approved master and are **identical across all 256 pixels (0 RGB differences)**. The normalized output's alpha channel is uniformly 255 (the normalizer always writes RGBA; the master itself has no alpha channel to compare, so alpha-uniformity is checked in isolation rather than round-tripped).

The two files are not byte-identical at the file level (SHA-256 differs: master is PNG colorType 2 / 3 bytes-per-pixel, normalized output is PNG colorType 6 / 4 bytes-per-pixel — an encoding difference, not a pixel-data difference), which is why this section proves pixel-level identity directly rather than relying on a file-hash match (matching `dirt_center`'s own precedent, whose master already had an alpha channel and so did produce a hash-comparable pixel layout — this is the first target where the master's colorType differs from the normalizer's output colorType).

## 6. Seam, repetition, palette, and motif audit

All figures below were **independently computed** directly from the normalized 16×16 RGBA pixel data (not copied from ChatGPT's runtime-audit report), then compared against the reported figures as a cross-check.

### Visual inspection

- **Comparison panel / 3×3 / field previews**: calm, low-contrast mottled blue water texture; no hard grid lines at any tile boundary.
- **No baked objects**: no lilies, reeds, rocks, or shore anywhere in the 16×16 result — pure water texture, matching the spec's "Water-base tiles must contain only water texture and subtle highlights" rule.
- **Expected single-tile repetition**: because only `water_a` exists, the 3×3 and 12×8 previews show the *same* 16×16 tile repeating at a fixed period — expected, and exactly why `water_b` and the shimmer frames exist for later work (out of scope here).

### Deterministic edge/wrap check

| Metric | Independently computed | ChatGPT-reported |
| --- | --- | --- |
| Average column-wrap step (col 15 → col 0) | 10.48 | — |
| Average internal column step | 9.09 | — |
| **Wrap ÷ internal ratio (horizontal)** | **1.153** | 1.153 |
| Average row-wrap step (row 15 → row 0) | 10.88 | — |
| Average internal row step | 10.83 | — |
| **Wrap ÷ internal ratio (vertical)** | **1.004** | 1.004 |

Both ratios match ChatGPT's reported figures exactly. A seam would show up as the wrap step being *much larger* than the typical internal step (an abrupt jump not seen anywhere else in the texture, typically several times larger). Both ratios are close to 1.0 (within ~15%), consistent with normal texture-noise variation, not a seam signature.

### Directional brightness gradient check

| Metric | Independently computed | ChatGPT-reported |
| --- | --- | --- |
| Row-luminance range | 9.62 (≈11.8% of mean) | — |
| Row-luminance correlation with row index | **-0.157** | -0.157 |
| Column-luminance range | 4.76 (≈5.9% of mean) | — |
| Column-luminance correlation with column index | **0.378** | 0.378 |

Both correlations match exactly. Neither is close to ±1, meaning the luminance variation across the tile is textural noise/mottling, not a monotonic lightening/darkening trend that would expose repetition. The column correlation (0.378) is the larger of the two — moderate but well short of a directional-gradient signature — consistent with the spec's note that "mild fixed-period mottling remains expected until `water_b` and shimmer frames exist" to break up the single-tile repeat.

### Palette compatibility (`arcane`/`water` + `forest` families)

`farm_environment_palette_v1.json` locks the `water` hex family (which `arcane` maps to for this target, per `contractFamilyMapping`'s "cool water range only" note) to `["#0B4961", "#0A6089", "#3E9DC6"]` and `forest` to `["#0A3521", "#174F1D", "#325E19", "#427118", "#6C8B15", "#91A513"]` — the two families this target's spec declares (`paletteFamilies: ["arcane", "forest"]`). For every one of the 256 pixels in the normalized tile, the nearest-swatch RGB Euclidean distance (across both families combined) was computed:

| Metric | Independently computed | ChatGPT-reported |
| --- | --- | --- |
| Pixels within 40 units of a locked swatch | **253 / 256** | 253/256 |
| Median distance | **19.38** (true statistical median of the 256 sorted distances) | 19.4 |
| Maximum distance | **48.22** | 48.2 |
| Minimum distance | 4.90 | — |

All figures match (the median's 19.38 vs. 19.4 and max's 48.22 vs. 48.2 are simple rounding of the same underlying value). Restricting the check to the `water` family alone produces identical figures (253/256, median 19.38, max 48.22) — every off-family pixel is already closer to a `water` swatch than a `forest` swatch, so the `forest` family contributes no additional coverage for this water-dominant tile, as expected. The candidate reads as natural shading/mottling of the locked water palette, not off-family color.

### Additional deterministic checks (independently added, not part of ChatGPT's reported figures)

Run as extra verification given this is the project's first water-terrain candidate and the spec explicitly calls out several specific artifact shapes to avoid ("no diamond lattice, scales, ripple rows, rings, checkerboard, or hard seam"):

- **Checkerboard/parity scan**: mean luminance of even-parity pixels (81.50) vs. odd-parity pixels (81.00) — a 0.5 gap, negligible; a real checkerboard artifact would show a large, consistent parity split. **No checkerboard signature.**
- **Plus/cross motif scan** (the same deterministic method used for `dirt_center`'s flagged defect: interior pixels where all four axis-neighbors differ from the center by a large, similar delta): **1 out of 196 interior pixels** matched — consistent with random mottling noise, not a systematic motif. **No plus/cross artifact.**
- **Row-repetition (ripple-row) scan**: compared every pair of rows for near-duplication; the most similar pair (rows 3 and 9) still differs by an average of 6.94 RGB-distance units per pixel, well above a near-duplicate threshold (which would read well under 3) and comparable to the internal row step (10.83). **No repeating ripple-row band.**

### Format checks

- No JPEG artifacts possible: master, canonical source, and normalized output are all PNG (lossless) end-to-end.
- No unexpected alpha: normalized output alpha channel is uniformly 255 (fully opaque) across all 256 pixels.
- Runtime smoothing: not applicable at this stage — no runtime integration occurred (out of scope for this task). The project-wide engine setting is nearest/point sampling (`VISUAL_ASSET_CONTRACT.md` §3); this was not independently re-verified here since nothing was loaded into Phaser.

## 8. Pipeline note for the next author

`scripts/upscale-nearest-neighbor.mjs`'s read stride is now unconditionally 4 bytes/pixel, matching `readPng()`'s always-RGBA-expanded output for both colorType 2 (RGB) and colorType 6 (RGBA) sources.
